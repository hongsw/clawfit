# Reference layers vs adoption levels

`clawfit` now uses two different but complementary maps.

## 1. Reference layers
These answer:
- what parts of the ecosystem exist?
- what category does a repo, product, or pattern belong to?

This is the domain of:
- runtimes
- harnesses
- team workflow systems
- capability layers
- research/eval layers
- data/knowledge infrastructure
- interface layers
- methodology signals

See: [reference-levels.md](../reference-levels.md)

## 2. Adoption levels
These answer:
- how far has a person/team/organization actually gone in integrating AI into work?
- are they at no-AI, prompting, harness, orchestration, or methodology-native stages?

See: [adoption-levels.md](./adoption-levels.md)

## Why both are necessary
If we only track tools, we miss the maturity of usage.
If we only track maturity, we miss what structural options exist in the market.

So the two maps work together:
- **reference layers** explain the outside world
- **adoption levels** explain the internal maturity of use

## Simple mental model
- Reference layers = **ecosystem structure**
- Adoption levels = **human/team maturity**
