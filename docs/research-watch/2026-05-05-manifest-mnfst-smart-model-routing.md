# Research Watch: Manifest (mnfst) — Self-Hosted Smart Model Routing for Agents

- Repo: https://github.com/mnfst/manifest
- Also see: https://github.com/mnfst/openclaw-model-router (Manifest skill for OpenClaw, 6★); https://manifest.build/ (product site); https://mnfst-manifest.mintlify.app/ (official docs); prior art cluster: LiteLLM, Portkey, OpenRouter (cloud-proxy model), Martian (ML-based routing)

## Why this is worth watching

Manifest crossed 6,095 stars with +122/day velocity (GitHub Trending TypeScript #7, 2026-05-05), placing it comfortably above the 5k-star clawfit promotion threshold on day of signal capture. It represents the third cost-arbitrage routing signal in 12 days — after Sub2API (2026-04-28) and DeepClaude (2026-05-04) — but with a meaningfully different structural claim: those two route through a single alternative backend, while Manifest routes dynamically across 300+ models at request time based on complexity scoring. The 269-release cadence (v6.0.1 as of 2026-05-01) and a stable NestJS + PostgreSQL production stack distinguish it from alpha-quality tools in this cluster.

## What stands out immediately

- **Two-layer routing architecture, stated explicitly in CLAUDE.md.** Layer 1 (always active): 23-dimension keyword analysis assigns each request to one of four complexity tiers — simple, standard, complex, reasoning. Layer 2 (optional): nine specificity categories (coding, web_browsing, data_analysis, etc.) detected via keyword analysis and tool heuristics. Resolution order: specificity check → complexity scoring → tier assignment → provider resolution → proxy forward. The 23-dimension claim is a concrete architectural figure, not marketing copy, and is verifiable in the routing source (claim to inspect until confirmed by independent code audit).
- **Sub-2ms routing decision latency (claim to inspect).** The product documentation and secondary sources consistently cite "under 2ms" for the routing algorithm. No independent benchmark has confirmed this figure. The routing algorithm operates over keyword scoring rather than ML inference, which makes the claim plausible — but no controlled test has been published comparing routing-added latency against a direct passthrough baseline.
- **Pricing data sourced from OpenRouter API, fetched daily.** This is architecturally significant: Manifest does not hardcode model pricing. It pulls live rates from OpenRouter to determine the cheapest model in each complexity tier from the user's connected providers. This means Manifest inherits OpenRouter's pricing coverage while positioning itself as free and self-hostable — a direct structural counter to OpenRouter's 5% per-request fee.
- **Self-hosted Docker deployment is the primary differentiator.** `bash <(curl ... install.sh)` brings up a full NestJS + PostgreSQL + SolidJS UI stack. Cloud option (app.manifest.build) exists but the GitHub issue #1589 discussion explicitly frames self-hosted routing as the category-defining position against cloud proxies. This matters for `data_sensitivity: confidential` profiles — no request payload leaves the operator's network.
- **Multi-tenant data model: User → Tenant → Agent → AgentApiKey.** The hierarchy is designed for teams managing multiple agents under one routing plane. This is a structural indicator of L2 or L3 relevance beyond individual developer use.
- **"70% cost reduction" claim is unqualified (claim to inspect).** The claim appears in repo description, docs, and secondary coverage uniformly, but no methodology is published. Which model mix? What request distribution? What quality benchmark was used to define "zero quality loss"? The figure may be accurate for specific workload shapes (many simple queries routed to GPT-4o-mini instead of Claude Opus) but cannot be taken at face value across all use cases.
- **org identity ambiguity: BaaS heritage.** The mnfst org hosts `docs-baas` and `website-baas` repositories — a separate legacy Backend-as-a-Service product (single YAML file backend). The 6k-star `manifest` repo is now unambiguously the model routing product, but the name overlap may create indexing noise. Teams evaluating Manifest should confirm they are looking at the routing product, not the older BaaS surface.
- **OpenClaw integration is primary; other agents via generic OpenAI-compatible endpoint.** The `openclaw-model-router` skill (6★) is the named first-party integration. Hermes, Vercel AI SDK, LangChain, and standard OpenAI SDK clients work via generic `/auto` endpoint passthrough. MCP integration is not mentioned in current documentation.

## Why clawfit should care

Manifest fits precisely into the multi-vendor anti-lockin meta-pattern confirmed 2026-04-28 (cc-switch, Sub2API, cmux). However, it occupies a distinct layer: where cc-switch switches the entire agent's provider configuration and Sub2API pools subscriptions, Manifest operates transparently per-request — the agent does not change behavior, the routing plane decides which model receives each call. This is a different architectural locus, closer to an inference middleware than a configuration manager.

For clawfit's scoring model, Manifest's existence exposes a gap: the current LLM preference weight (0.15) and cost weight (0.25) assume the agent-to-LLM binding is fixed for a given recommendation. A Manifest-fronted agent breaks that assumption — the effective LLM is probabilistically distributed across the user's provider set at inference time, not a fixed registry entry. A user running Claude Code + Manifest is not the same as a user running Claude Code directly; their cost profile and latency profile are shaped by the routing tier mix.

The `budget` filter in clawfit (`--budget 0.01`) currently applies to a specific LLM's per-token price. If Manifest is in the stack, the effective per-token cost is a blended rate that clawfit cannot currently express. This is a concrete schema gap worth flagging for the next calibration cycle.

## Preliminary interpretation

Current best reading:
- **Level 4c — Tool / MCP / capability layer (model-routing gateway sub-type)** (primary). Manifest sits between an agent and its LLMs — it is not itself an agent runtime (L1), not a harness controlling agent workflow (L2), not a governance layer (L3), and not a memory or context system (L5). It augments agent capability by transparently optimizing the inference call. The closest existing L4c analogy is Sub2API (subscription-pooling gateway), but Manifest's per-request routing intelligence distinguishes it from a static proxy.
- Secondary L2 read is weak but present. The multi-tenant hierarchy (Tenant → Agent → AgentApiKey) and team-level cost tracking/alert architecture suggest the product is beginning to build upward toward harness-adjacent concerns. Not sufficient for L2 primary today; watch for agent lifecycle management features in future releases.
- **Not L2 primary.** Manifest does not control agent workflow, task decomposition, sprint contracts, or execution sequencing. It only intercepts the LLM call. Classifying it L2 would conflate a request-level gateway with a workflow orchestrator.
- **Competitor cluster for L4c sub-type.** LiteLLM (Python-native, widely deployed), Portkey (cloud-first observability + routing), OpenRouter (cloud proxy, 5% fee), Martian (ML-based routing with quality guarantees). Manifest's differentiation vector: self-hosted, no per-request fee, OpenClaw-first integration, built-in spending limits + alerts.

## Status

Hold for one round: (1) confirm the "sub-2ms routing decision" claim via independent benchmark or source inspection; (2) confirm the "70% cost reduction" figure has a published methodology; (3) verify whether MCP server support is planned or present (would strengthen L4c position).
