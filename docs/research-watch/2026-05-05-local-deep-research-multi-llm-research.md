# Research Watch: local-deep-research — LearningCircuit

- Repo: https://github.com/LearningCircuit/local-deep-research
- Also see: https://github.com/karpathy/autoresearch (L5 anchor, 77k★); https://github.com/huggingface/ml-intern (institutional research-loop signal, 2026-04-24); https://github.com/langchain-ai/open_deep_research (LangChain research-loop comparator)

## Why this is worth watching

local-deep-research has reached 4,841★ with +171/day velocity (GitHub Trending Python #9, 2026-05-05) — approaching the 5k registry threshold with no signs of slowing. It occupies a distinct position in the L5 research-loop cluster: unlike karpathy/autoresearch (solo researcher, conceptual) or huggingface/ml-intern (ML-domain-specific), LDR is a self-hosted web application targeting general knowledge work across academic and consumer use cases, with explicit offline/local-LLM support as a primary design constraint rather than a configuration option. The LangGraph agent strategy (autonomous engine-selection, adaptive mid-research routing) and a shipped MCP server (8 tools, STDIO transport) make this the first research-loop tool in the taxonomy with both a production-grade agentic mode and a first-class MCP integration.

## What stands out immediately

- **Agentic research mode via LangGraph.** The "LangGraph agent strategy" is the key architectural distinguisher from pipeline-based research tools: the LLM determines which search engines to query, switches between them based on findings, and decides when synthesis is warranted — as opposed to a fixed query → fetch → summarise sequence. This is an agent control loop inside a research tool, not a research workflow inside an agent harness. Claim-to-inspect: documentation describes this as "significantly more sources than pipeline-based strategies" and "adaptively switches," but no independent comparative benchmark confirms the source-count or quality claim.
- **10+ search engine integrations across academic and general sources.** arXiv, PubMed, Semantic Scholar, NASA ADS, Zenodo (academic); Wikipedia, SearXNG, GitHub, Wayback Machine, The Guardian (general); Tavily, SerpAPI, Brave (premium). The academic-sources cluster (5 distinct engines) is the widest in any L5 research-loop tool currently in this taxonomy. This differentiates LDR from web-only research agents and directly serves `task: research` profiles in science, healthcare, or engineering contexts.
- **Multi-LLM backend support.** Local: Ollama, LM Studio, llama.cpp. Cloud: OpenAI, Anthropic, Google, 100+ via OpenRouter. Explicit `network: offline` support via any Ollama-compatible model — the research tool works with zero cloud dependency when paired with a local model. This is structurally important for clawfit's `data_sensitivity: confidential` profiles.
- **MCP server with 8 tools (5 research, 3 discovery).** Exposes `search`, `quick_research`, `detailed_research`, `generate_report`, and `analyze_documents` to any MCP-compatible client (Claude Desktop, Claude Code). STDIO transport only — explicitly "designed for local use only" and "do not expose over a network without proper security controls." This is an important limitation: LDR's MCP surface is a local-context tool, not a remotely composable service. The tool set bridges L5 (research loop) and L4c (MCP action) but the security constraint limits practical L4c use to local setups.
- **~95% on SimpleQA benchmark — claim requires independent verification.** The README cites "~95% accuracy on SimpleQA benchmark" with GPT-4.1-mini + SearXNG (focused-iteration strategy). The documentation itself flags this as "preliminary results from initial testing" and "limited sample size." SimpleQA favors factual short-answer retrieval, which strongly rewards a multi-engine search approach — the architecture is a good fit for this benchmark. Treat this as a structural directional claim, not a validated performance tier. Community-maintained leaderboards on Hugging Face and GitHub track model + engine + strategy combinations but no third-party confirmation of the headline number has been identified.
- **Self-hosted web app with REST API, WebSocket, and export.** Runs as a Docker Compose service; encrypted per-user SQLite database (SQLCipher/AES-256); REST API with CSRF auth; WebSocket for real-time progress; PDF, Markdown, JSON, HTML export. This is a production deployment posture unlike most L5 research-loop tools (which are typically CLIs, notebooks, or library APIs). The analytics dashboard (cost tracking, token usage, response times) suggests real-world budget-management use rather than purely research/experimental use.
- **20+ research strategies; 30+ search modes.** Quick Summary (30s–3min), Detailed Research, Report Generation, Document Analysis, plus "source-based," "focused-iteration," "rapid," "parallel," "dual-confidence," "evidence-based," and others. The strategy vocabulary is not standardised across the field; these are internal naming conventions, not interoperable APIs. However, the breadth signals a mature feature surface relative to most L5 tools.
- **Python SDK + community leaderboard on Hugging Face.** The LDR Python SDK (`from local_deep_research.api import LDRClient, quick_query`) allows programmatic integration. The Hugging Face leaderboard aggregating model + engine + strategy performance is an unusual infrastructure signal for a 4.8k-star tool — typically leaderboards at this scale are initiated by well-resourced institutional projects. The leaderboard's methodology has not been independently reviewed.
- **Privacy architecture as a primary constraint, not a feature toggle.** No telemetry, no analytics, no phone-home calls. Credentials in plaintext during active sessions — the documentation's acknowledgment that "this is industry-wide accepted reality" is accurate but worth noting: other local research tools with similar claims (e.g., self-hosted Perplexity alternatives) have varied in how thoroughly they enforce the no-phone-home claim. Claims here cannot be fully validated without source review.

## Why clawfit should care

The `task: research` path in clawfit's recommendation engine currently surfaces L1/L2 general agents (Claude Code Routines, OpenClaw, Refly) rather than dedicated research-loop tools. local-deep-research would be the first purpose-built research application in the registry if it clears 5k stars — and at +171/day, that is a near-term event. Its multi-LLM backend positions it as hardware-agnostic: it scores across `hardware: cloud`, `hardware: laptop`, and `hardware: workstation` depending on the underlying inference backend. The `network: offline` + `data_sensitivity: confidential` combination is directly addressable via an Ollama-backed local deployment, filling a gap that no current registry entry covers for dedicated research use cases.

The MCP server integration is also structurally relevant: if LDR surfaces as a recommended tool, clawfit may need to note the STDIO-only constraint as a deployment condition (the tool is composable within a local agent context but not usable as a remotely-deployed research capability). This is a new kind of constraint not currently expressed in the registry schema.

One schema gap surfaces: `task: research` is treated as a generic label in the current scoring model. LDR's academic-source cluster (arXiv, PubMed, Semantic Scholar) serves a researcher profile that is distinct from the `large_exec_research` persona currently tracked (which maps to general executive decision support, not domain academic work). If LDR reaches the registry, a `task: academic-research` or `data_domain: scientific` axis would sharpen recommendation precision.

## Preliminary interpretation

Current best reading:

- **Level 5 — Research / evaluation / autoresearch (primary):** local-deep-research is a direct structural sibling to karpathy/autoresearch (the L5 anchor). Both are multi-source research-loop tools that wrap LLMs with iterative search + synthesis pipelines. LDR is substantially more production-oriented (web app, Docker, REST API, user auth, encrypted DB, analytics) and community-maintained rather than solo-researcher provenance. It extends the L5 cluster with: (a) self-hosted web-app deployment, (b) the widest academic-source integration in the taxonomy, (c) an autonomous agentic mode via LangGraph, and (d) a local-first, multi-LLM-backend posture. No competing classification is stronger — the tool's purpose is to conduct research on behalf of users, not to provide memory or capability to an existing agent session.
- **Level 4c — MCP tool-use layer (secondary, constrained):** The LDR MCP server exposes research as an action surface to Claude Desktop and Claude Code. This is a secondary surface, not the primary classification: the MCP interface is a consumption layer on top of the research application, and the STDIO-only / local-only transport constraint limits it to single-machine deployments. If the MCP server were exposed as a cloud-hosted endpoint (against current documentation guidance), a stronger L4c read would be warranted.
- **Not Level 2.** LDR does not orchestrate or wrap existing agent runtimes; it is a self-contained application that uses LLMs as a component. The LangGraph agent strategy is internal to the research pipeline, not an exposed harness for external agents to use.
